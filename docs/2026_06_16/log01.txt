## 一、项目背景与定位

**双重定位**：
- 一款类 Hacknet 的命令行风算法解谜游戏
- 一门 Java 课程的课程设计作业，需体现"严肃工程感"

**核心玩法**：玩家在每关被给予一组数据结构对象、一组允许使用的指令（带次数配额）、一个起始状态和一个目标状态，需在有限步数内通过输入指令将系统从起始变换到目标。

**技术挑战**：
- 玩家可输入的"指令"形式自由，但又要让引擎能解析、验证、补全
- 需支持运行期动态扩展结构与指令
- 需要面向 AI 的语义描述层，让 LLM 能理解局面并给出提示
- 教育意义：代码应展现良好工程实践

### 2.2 状态机模型

整个后端是单一状态机，对外只有一个入口：

```java
public static String[] Logic.interact(String command)
```

返回值始终是单元素数组，内容是 JSON 字符串。所有玩家交互、所有查询都走这一个接口。

**状态划分**：
- **全局状态**（与进程同生命周期）：存档 Progress、关卡树浏览 LevelTree
- **关卡状态**（与一关同生命周期）：LevelState 容器，含 RuntimeContext、LevelConfig、指令索引等

**指令分三类**：
- 全局指令：任何时候可用（`query:rowbuffer`、`action:ResetSaveData`）
- 选关指令：未在关卡内时可用（`query:browse`、`action:start:<n>`、`action:back`）
- 关卡内指令：已开始一关时可用（`action:input:<token>`、`query:nextcommandpart`、`query:objects`、`query:quota`、`query:levelinfo`、`query:aiContext`）

### 2.3 数据流：玩家提交一条语句

```
action:input:<char>* → 累积到 inputBuffer
                ↓
       query:nextcommandpart 实时反查补全
                ↓
       action:input:enter 提交
                ↓
       StatementExecutor.execute（玩家身份）
         ├── extractStructId       O(1) 定位结构
         ├── PatternMatcher        括号感知提取参数
         ├── ArgHintResolver       语义校验
         └── Template.executeCommand 真正执行
                ↓
       结构方法内部可能 triggerEngineCommand（系统身份连锁）
                ↓
       LevelLoader.executeJudgeCommands
                ↓
       检查 isWinConditionMet
         ├── 胜 → LevelTree.applyClear + SaveManager.save + 退回父层级
         ├── 步数耗尽 → fail + 退回父层级
         └── 否则 → 继续，currentStep++
```

## 三、关键设计巧思

### 3.1 Template / Instance 分离

**问题**：早期设计中，每个结构（`FakeStack`）的构造函数既负责加载该结构的所有指令（反射、注册），又被运行期的指令（`Copy`、`InitEmpty` 等）当成游戏数据对象的构造器使用。结果每创建一个游戏对象就把整个指令注册流程跑一遍。

**解法**：用静态内部类把两种角色分开。

```java
public class FakeStack extends Abstract {
    // ============ Template 角色 ============
    // 由 Logic.registerStructures() 调用一次
    // 负责反射加载所有指令
    public FakeStack() {
        super();
        this.methodRegistry.putAll(MethodRegistryLoader.load(STRUCTURE_ID));
        ifLoadMethodDynamically("init_full");
        // ...
    }

    // ============ Instance 角色 ============
    // 由游戏指令在运行期 new
    // 只持有数据，无任何反射逻辑
    public static class Instance extends Abstract.Instance {
        public int[] array;
        public int top;
        // ...
    }
}
```

`RuntimeContext.objects` 存的全是 `Abstract.Instance`，Template 实例只在 `Logic.structidToStructure` 持有。这彻底解决了"指令注册与游戏数据创建混杂"的问题。

### 3.2 括号感知的 Pattern 匹配

**问题**：玩家输入的语句可能含嵌套括号（`Command.in[(Queue[(A)].pop)]`），简单的"找下一个字面量"无法正确提取参数。

**解法**：`PatternMatcher` 引入括号深度感知。`@` 在 pattern 中对应输入里的 `(...)`：

```
pattern "Queue[@].pop"        匹配 "Queue[(A)].pop"     → args=["A"]
pattern "Command.in[@]"       匹配 "Command.in[(Queue[(A)].pop)]"
                                                       → args=["Queue[(A)].pop"]
```

**实现核心**：`findMatchingClose(s, openIdx)` 从 `(` 出发维护深度计数，深度归零时返回对应 `)` 下标。`[` 和 `]` 是字面量，不参与深度计算——只有 `(` `)` 是定界符。

**三个对外方法**：
- `extractArgs(def, statement)` — 完整匹配并提取参数
- `couldBePrefix(def, input)` — 判断是否可能是合法语句的前缀
- `nextCandidates(def, input)` — 返回补全候选（字面量片段 + argHint 槽位）

### 3.3 可扩展的 ArgHint 语义系统

**问题**：每个 `@` 位置可能要求不同语义——某个对象、某种指令、自由字符串。怎么统一表达并允许将来扩展？

**解法**：每个 `StructureMethod` 声明 `String[] getArgHints()`，每个元素是一个语义描述符：

```
"any"                    — 自由字符串
"obj"                    — 任意存活对象
"obj[Queue,Stack]"       — 指定结构的存活对象
"cmd"                    — 任意已注册指令
"cmd[Stack]"             — 某结构的所有指令
"cmd[Stack.pop]"         — 某结构的具体指令
"cmd[in]" / "cmd[out]"   — 按标签筛选
```

`ArgHintResolver` 用**策略模式**分发：

```java
private static final ArgHintSpec[] SPECS = {
    new AnySpec(), new ObjSpec(), new CmdSpec()
};
```

每种 hint 一个 Spec，实现 `accepts / enumerate / completions / validate`。将来加 `cmd[in][Queue]` 这种新语法只需新增一个 Spec 并注册进数组，主流程不动。

### 3.4 递归词法补全（含配额上下文切换）

**问题**：`cmd[...]` 类型的 `@` 位置内部又是一条完整指令，玩家在嵌套处输入时也要有补全。同时**外层指令受配额限制，内层语义层不受**——外层是"玩家有没有权限用"，内层是"这条指令在语法上是不是合法"。

**解法**：`ArgHintResolver.completionsForDef(def, input, checkQuota, ...)` 把"是否检查配额"做成参数贯穿递归。

```java
最外层 Lexer.compute(state)
    → completionsForDef(def, current, checkQuota=true, ...)
        ├── 字面量片段 → 加入选项
        └── argHint 片段
            → dispatch(hint).completions(...)
                → CmdSpec 内部对每条候选 def 调
                    completionsForDef(def, soFar, checkQuota=false, ...)
                        ← 递归，且配额检查关闭
```

外层"玩家有 6 次 pop"，内层"凡是已注册的 pop 都算合法"——两套世界自然解耦。

### 3.5 关卡解锁的"前置 + 跳过"系统

**问题**：早期想法是"通关一关解锁固定几关"，但用户很快意识到这不能表达"任意打通 3 个魔王中的 2 个就开门"。

**解法**：每个节点声明 `requires`（前置 index 列表）和 `skippable`（可跳过数量）。存档只存 `cleared`，**不存** `unlocked` —— unlocked 是计算量极小的纯函数：

```java
boolean isUnlocked(NodeEntry e, Set<Integer> doneSet) {
    if (e.requires == null || e.requires.isEmpty()) return true;
    int need = e.requires.size() - Math.max(0, e.skippable);
    if (need <= 0) return true;
    int done = 0;
    for (Integer req : e.requires) {
        if (doneSet.contains(req)) {
            done++;
            if (done >= need) return true;
        }
    }
    return false;
}
```

**冒泡**：层级也可声明 `clear_when` + `clear_when_skippable`——子层级满足条件后，本层级被视为父层级中的一个 cleared 节点，向上递归直到不再满足。这一套机制同时表达了"AND"、"N 选 K"、"层级冒泡"三种解锁模式。

**隐藏关**：节点加 `hidden: true`，未解锁时不出现在 browse 中。

### 3.6 全局静态行缓冲（System.out 模式）

**问题**：调试日志要让所有代码"无感知"地调用，又要能在前端按需查询。

**解法**：`RowBuffer` 仿 `System.out` 的无感知调用方式。**双重检查锁**单例 + **静态门面**：

```java
RowBuffer.append("...");           // 任何代码任何位置都能写
RowBuffer.getRecent(10);            // 取最近 10 条
RowBuffer.getRecent();              // 取全部
RowBuffer.getUnread();              // 消费型查询
```

底层是带容量的 `LinkedHashMap`，超容量自动淘汰最老条目。设计上接近"环形缓冲 + 自增序列号"。

### 3.7 慢加载的 AI 描述系统

**问题**：要给 AI 描述每个结构和指令的特性，但描述文本如果硬编码在 Java 类里，文案修改要重新编译，且让代码失去"纯逻辑"的感觉。

**解法**：完全数据驱动 + 慢加载缓存：

```
resources/structure/
├── structureDescription.json        ← 结构特性，全局只读一次
└── queue/methodDescription.json     ← 指令描述，按结构粒度缓存
```

`AiDescription` 是 process-wide 单例，`forStructure(id)` / `forMethod(id, cmd)` 触发懒加载，加载失败静默兜底返回空串。`ContextBuilder` 拼字符串时段落为空就跳过——文案缺失不影响功能可用。

预加载入口 `ensureLoaded(structIds)` 在 `LevelLoader` 装填末尾调用，把当前关卡用到的结构一次性加载完。

### 3.8 受限的存档系统

**问题**：存档要落盘、要支持 ResetSaveData，但安全性必须保证（不能让攻击者通过路径注入写到任意位置）。

**解法**：`SaveManager` 把 `SAVE_DIR` 固定为相对工作目录的 `saves/`，每次写入前都校验：

```java
Path target = SAVE_FILE.normalize();
if (!target.startsWith(SAVE_DIR)) {
    RowBuffer.append("[Save] 拒绝越权写入: " + target);
    return false;
}
```

只有这一个写入路径常量，文件名也是固定的 `save.json`，无法被外部输入影响。**绿色可迁移**：相对路径意味着整个安装包可以随意移动，存档跟着走。

### 3.9 三角分工的状态机拆分

**Logic 拆分前**：单类 595 行，分发、装填、执行、词法、输入处理、浏览全在一起。

**拆分原则**：把"数据"、"算法"、"协调"三种职责物理隔离：

- **数据**（`LevelState`）：纯字段容器，可被任意模块读写
- **算法**（`lang/` 下五个类、`Lexer`、`InputHandler` 等）：无状态静态方法，接收 LevelState 出入
- **协调**（`Logic`）：单例门面，负责跨模块调用

**反向调用约定**：算法模块如果需要触发全局副作用（通关回写、启动新关），通过 package-private 方法回调 Logic（`logic.handleLevelClearedExternal()` 等）。这样依赖图保持单向，只有少数协调点反向。

**结果**：Logic 252 行，其余模块都在 100~200 行之间，依赖图无环。

### 3.10 「过拟合」AI 助手的纯事实接口

**问题**：AI 提示如何接入？后端要不要内嵌 LLM 客户端？

**解法**：后端只产出"事实文本"，不绑定任何 LLM 提供方。`query:aiContext` 返回一段 Markdown 字符串，包含：

- 游戏总规则（硬编码）
- 当前关卡用到的结构特性（数据驱动）
- 初始状态、目标状态、关卡作者私下留给 AI 的提示（关卡 JSON 中的 `ai_hint` 字段，玩家不可见）
- 当前局面（对象快照、buffer 配置、剩余步数）
- 玩家仍可用的指令清单（含剩余次数、参数语义）

**不暴露**判分指令（避免泄露答案）、**不附加** prompt 工程指令（怎么用是前端的事）。

前端拿这段文本拼到自己的 prompt，喂给任意 LLM（OpenAI / DeepSeek / 本地模型都行）。后端零依赖、零适配代码。

## 四、模块详解

### 4.1 lang/ — 词法与执行（无状态算法）

| 类 | 职责 |
|---|---|
| `PatternMatcher` | 括号感知的 pattern 匹配、前缀判断、补全推进 |
| `ArgHintResolver` | argHint 语义展开（obj/cmd/any）、递归补全、参数校验 |
| `Lexer` | 顶层词法分析，遍历所有有配额的 def，汇总补全选项 |
| `StatementExecutor` | 语句执行，匹配 def → 校验语义 → 调用结构方法 |
| `CommandDefinition` | 指令描述符（pattern、argHints、tags、配额计数） |

### 4.2 logic/ — 状态机协调

| 类 | 职责 |
|---|---|
| `Logic` | 单例门面，唯一对外 `interact()` 入口，dispatch 路由 |
| `LevelState` | 关卡级数据容器（runtimeContext、levelConfig、三张指令索引等） |
| `LevelLoader` | 关卡装填的五个阶段（load → register → applyLimits → buildTagIndex → init） |
| `InputHandler` | 处理 input token（字符 / tab / del / enter / up / down / exit） |
| `Browser` | LevelTree 数据 ↔ browse 响应的适配层 |

### 4.3 protocol/ — 对外协议

| 类 | 职责 |
|---|---|
| `ResponseBuilder` | 统一 JSON 响应打包，所有 `interact()` 返回都走这里 |
| `ContextBuilder` | 装配 `query:aiContext` 的 Markdown 文本 |

### 4.4 structure/ — 数据结构域

每个结构包含一个 Template 类和一个 method 子包。当前已实现：

- **Stack**：8 个指令（initEmpty / initFull / copy / delete / equal / pop / push）
- **Queue**：7 个指令（同上但用 add 替代 push）
- **PQ**：8 个指令（多 initFullConfig 用于自定义大/小根堆和去重）
- **Command**：3 个元指令（clearBuffer / in / out）

### 4.5 tools/ — 横切设施

| 包 | 职责 |
|---|---|
| `buffer/` | 全局调试日志 RowBuffer |
| `aidescription/` | AI 描述文本的慢加载缓存 |
| `jsonloader/` | POJO（`model/`）+ 加载器（`loader/`） |
| `leveltree/` | 关卡树浏览、解锁判定、通关冒泡 |
| `savedata/` | Progress 数据模型 + SaveManager 安全落盘 |

## 六、演进历程与设计决策

### 阶段一：从循环死锁到状态机
- 早期是单线程 `while(true) { read; execute; }` 循环，无法被外部驱动
- 重构为单例状态机，外部通过 `interact(String)` 单点驱动
- 每次调用都是独立交易，无内部阻塞

### 阶段二：JSON 协议化
- 早期返回纯文本（"[OK] 关卡已加载"）
- 改为统一 JSON 协议 `{ok, type, data}`，所有响应都通过 ResponseBuilder
- 前端用 `ok` 字段判断成败，`type` 字段判断业务语义

### 阶段三：语法演进
- 早期 pattern 用 `Queue(@).pop`，括号是字面量
- 改为 `Queue[@].pop`，括号成为参数定界符，支持深度嵌套
- `ArgHintResolver` 加入语义层，让补全候选动态过滤

### 阶段四：词法系统的递归化
- 早期补全只支持顶层字面量匹配
- 引入 `ArgHintSpec` 策略模式 + `completionsForDef(checkQuota)` 递归参数
- 支持 `cmd[...]` 嵌套场景下的内层补全

### 阶段五：关卡组织
- 早期是平铺的整数关卡号（`001.json`、`002.json`）
- 改为树状目录 + `nodeRegistry.json` + 解锁规则
- 「requires + skippable」机制统一表达 AND、N 选 K、跨层级冒泡

### 阶段六：Logic 拆分
- Logic 单类一度膨胀到 595 行
- 按"数据 / 算法 / 协调"三角拆分为 7 个模块
- 每个模块 100~200 行，依赖图无环

### 阶段七：AI 上下文层
- 加入 `query:aiContext` 装配 Markdown 文本
- `AiDescription` 慢加载从平行 JSON 文件读取描述
- 保持代码与文案的物理分离

每个阶段的演进都遵循同一原则：**先把不变性钉死，再做加法**。

## 七、对外协议

所有响应是 `String[]`，长度 1，内容是 JSON：

```
{ "ok": boolean, "type": string, "data": object }
```

### 7.1 响应类型表

| type | ok | 触发场景 |
|---|---|---|
| `ack` | true | 简单确认（已重置、已退出等） |
| `error` | false | 指令非法、参数错误、未解锁等 |
| `debug` | true | query:rowbuffer 调试日志 |
| `browse` | true | 选关界面信息 |
| `levelState` | true | enter 后仍在继续 |
| `win` | true | 通关 |
| `fail` | false | 步数耗尽 |
| `objects` | true | 存活游戏对象快照 |
| `commandHints` | true | 词法补全 |
| `quota` | true | 玩家配额状态 |
| `levelInfo` | true | 关卡信息 |
| `aiContext` | true | AI 上下文文本 |

### 7.2 输入指令分组

详见前面"指令集"表。设计原则：

- `query:` 前缀只读
- `action:` 前缀有副作用
- 大小写不敏感（`equalsIgnoreCase`）

## 八、可扩展性预留

### 新增数据结构
**只需**：写 Template 类（继承 `Abstract`）+ 写各方法类（实现 `StructureMethod`）+ 在 `structureRegistry.json` 注册 + 在 `methodRegistry.json` 注册指令 + 在 description JSON 加文案。**完全无需修改现有代码**。

### 新增 argHint 语法
比如未来想加 `cmd[in][Queue]`（同时按 tag 和按 struct 筛选），**只需**：在 `ArgHintResolver.SPECS` 数组中新增一个 `ArgHintSpec` 实现，主流程无需改动。

### 新增对外指令
在 `Logic.dispatch` 加 if 分支，在 `ResponseBuilder` 加打包方法。无需触动其他模块。

### 新增解锁规则
比如"通过 A 且通过 B 或 C"，**当前 `requires + skippable` 不能直接表达 OR**。要扩展的话改 `NodeEntry.requires` 为 `{all_of, any_of}` 对象形式，向后兼容数组形式即可。改动局限在 `LevelTree.computeUnlocked` 一个函数。

### 新增存档字段
`Progress` 类新增字段，Gson 反序列化对老存档自然兼容（缺字段填默认值）。

### 新增 AI 提示风格
`ContextBuilder.build` 是单个静态方法，可以多写几个版本（如 `buildVerbose`、`buildConcise`），新加 `query:aiContextVerbose` 之类的指令分发即可。

## 附录：关键设计断言

以下是项目中明确做出过的、贯穿后续所有迭代的设计断言：

1. **唯一入口**：所有交互必须通过 `Logic.interact(String)`
2. **统一协议**：所有响应必须是 JSON，必须经 `ResponseBuilder`
3. **状态机性质**：每次 `interact` 调用是独立交易，无内部阻塞
4. **Template / Instance 分离**：结构指令注册与游戏对象创建完全解耦
5. **代码 / 文案分离**：所有文案都在 JSON，代码不硬编码文案（仅 GAME_RULES 例外）
6. **数据 / 算法 / 协调分层**：算法模块无状态、数据模块只持有字段、协调集中在 Logic
7. **AI 中立**：后端不知道任何具体 LLM，只产出事实文本
8. **存档隔离**：所有文件操作只能在 `saves/` 目录
9. **加载即终态**：关卡装填完毕后，所有指令、对象、可见性都已就位，运行期不再变更指令注册表
10. **括号感知**：`(` `)` 是定界符，`[` `]` 是字面量，`@` 是参数槽——三者语义不重叠
