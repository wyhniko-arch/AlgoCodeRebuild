gradle build 执行后的产物性质分析

当在多模块根目录下执行gradle build时，Gradle会严格按照生命周期顺序，
并行或串行触发各子模块的compileJava、processResources、classes以及jar等一系列标准化Task。
最终的物理产物会严格输出到子模块的构建目录下：game-core/build/libs/game-core.jar。

在默认配置下，这个生成的game-core.jar是一个普通JAR文件（学专业术语：Plain JAR）。
它存在两个直接影响运行的缺陷：
缺少清单属性：其内部的META-INF/MANIFEST.MF文件中不包含Main-Class属性（即未指定谁是启动类）。
缺少依赖解包：它只包含game-core内部的编译字节码，并未将build.gradle中声明的Gson依赖库融合进去（并不是一个Fat JAR）。
因此，如果在终端中直接执行java -jar game-core/build/libs/game-core.jar，JVM大环境会直接抛出以下错误而拒绝运行：
no main manifest attribute, in game-core/build/libs/game-core.jar

解决方案：
有三种解决方案，一种是配置IDE语言服务器，一种是引入application插件，一种是通过类路径显式驱动。
那么第一种显然就是让IDE适应当前这个环境，我自己编译能过，但是其他开发者可能还得自己配，就很傻逼；
最后一种你不觉得很low吗？我都用构建工具了，最后一种纯纯脱离构建工具。
所以肯定用第二种方法。

日志中暴露了两个独立的问题。一个是导致程序崩溃的异常，另一个是终端编码错乱。
以下是针对这两个问题的工程定位。

1. 异常：java.util.NoSuchElementException: No line found
问题定位与原理解析
异常堆栈精确指向了 Core.java:370 的 scanner.nextLine()。
这是由于，当使用./gradlew :game-core:run启动程序时，Gradle并不是在当前终端直接运行代码，
而是分叉（学英语：这里的分叉也叫做Fork）出了一个独立的后台JVM进程来执行Main类。
在操作系统层面的默认安全与进程管理规范中，后台进程是无法直接读取宿主终端的标准输入流（即System.in）的。

从Core的视角来看，System.in处于一种“被切断”或“立刻到达文件尾（文件尾，没错，就是EOF）”的状态。
因此，当Scanner尝试读取下一行时，由于流中没有任何数据，直接抛出了该异常。

解决方案
在game-core/build.gradle中显式声明，将宿主终端的输入流桥接（学英语：Pipe）给Gradle运行的那个JVM进程。

2. 视觉污染：控制台中文乱码（锟阶讹拷一...）
问题定位与原理解析
日志中原本应该输出 [Debug] === 阶段一... 的地方变成了“锟斤拷”。这被称为“GBK 烫烫烫/锟斤拷锟斤拷”现象。
代码文件由于之前在工作区锁定了编码，是纯净的UTF-8。但是，在Windows系统的底层大环境中，
默认的活动代码页（学英语：Code Page）通常是 CP936 (即GBK)。

JVM在编译或运行时，默认采用了Windows环境的GBK编码去读取UTF-8源码或向控制台输出流，
由于字符映射表对齐失败，产生了乱码。

解决方案
必须在Gradle脚本中强约束Java编译任务以及JVM运行时的字符集变量，
使其无论在什么操作系统下，都绝对采用UTF-8。

接下来就是具体的操作。修改game-core/build.gradle，
打开现有的game-core/build.gradle，在文件最底部追加以下两个闭包配置。
这属于Gradle的Task级配置规范（是人话吗）：

// ==========================================
// 运行环境配置变量
// ==========================================
tasks.named('run') {
    // 【修复崩溃】将宿主终端的标准输入流 (System.in) 物理映射到运行中的 JVM 进程
    standardInput = System.in
    // 【修复乱码】强制运行期 JVM 向控制台输出时使用 UTF-8 编码
    jvmArgs = ['-Dfile.encoding=UTF-8']
}

// ==========================================
// 编译环境配置变量
// ==========================================
tasks.withType(JavaCompile).configureEach {
    // 【修复乱码】强制 Javac 编译器使用 UTF-8 解析源代码文件
    options.encoding = 'UTF-8'
}

这是由于输入流中断信号引发的运行时异常，进而传导至构建工具生命周期导致构建失败。
为什么Ctrl+Z会触发FAILED
1. 在Windows操作系统大环境中，在控制台输入Ctrl+Z并回车，相当于向当前进程的标准输入流发送一个EOF
（学英语：End-of-File，文件结束标记）信号，物理性地关闭了该输入流。
Scanner的标准行为约束：
    根据JDK官方规范，当Scanner.nextLine()在等待输入时遭遇流关闭（EOF），且缓冲区内没有整行数据时，
    其确定的硬性行为就是抛出java.util.NoSuchElementException: No line found。
2. Gradle的逻辑交互期间，standardInput = System.in变量配置成功生效，支持了游戏的正常输入。
键入Ctrl+Z则System.in被物理关闭。Core.java:370捕获到EOF则抛出NoSuchElementException异常。
由于该异常未被try-catch捕获，所以直接穿透至main线程顶部，导致JVM崩溃。

JUnitPlatform 的核心定义与作用
在现代Java开发中，JUnit Platform是JUnit 5架构的基石。简单来说，它是“测试执行引擎的底座”。
在以前的JUnit 3和JUnit 4时代，测试框架的API和执行引擎是强耦合在一起的。
开发工具（例如IntelliJ IDEA、Eclipse）或构建工具（例如Gradle、Maven）如果要运行JUnit测试，
必须硬编码去适配JUnit的特定版本。
为了解决这个耦合问题，JUnit 5进行了重构，提出了JUnit Platform的概念。

JUnit 5 = JUnit Platform + JUnit Jupiter + JUnit Vintage

JUnit Platform（平台层）：
这是最底层的基础设施。它不包含任何编写测试用例的注解（如 @Test），
它只定义了一套标准化的接口和启动规范（Launcher API）。
任何构建工具或IDE只要对接了JUnit Platform，就能拉起并运行所有适配了该平台的测试用例。
JUnit Jupiter（新引擎层）：
这是目前在JDK 21大环境下编写新测试所使用的标准框架（包含了 @Test, @BeforeEach, @ParameterizedTest 等新注解）。
它内部包含一个专门的测试引擎，挂载在JUnit Platform上运行。
JUnit Vintage（老引擎层）：
这是一个向后兼容的特殊引擎。它的作用是让JUnit 3和JUnit 4的老测试用例，不需要修改任何代码，
就能直接跑在全新的JUnit Platform上。

在终端执行 ./gradlew :game-core:test时，Gradle会启动其内部的测试任务。
如果不加useJUnitPlatform()：Gradle默认会降级使用老旧的JUnit 4执行器去寻找测试类。
因为引入的是JUnit 5（junit-jupiter），Gradle会直接报“找不到任何测试用例”（0 tests completed），
导致整个测试阶段被空跑。

由于JUnit Platform成了标准大环境，后续开发完核心引擎，测试类运行时，IDEA调用的是JUnit Platform。
通过Gradle命令行自动化构建时，Gradle调用的也是JUnit Platform。
这保证了过了编译后，测试用例在任何工具、任何CI/CD自动化大环境下的运行结果和行为是绝对一致的，
没有任何工具级别的环境差异。
