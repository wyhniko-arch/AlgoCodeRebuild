错误的大环境假设（前车之鉴）：
现象： 使用 ./gradlew :game-core:run 启动游戏时，中文出现乱码 ，且在终端输入 Ctrl+Z 退出时，
Gradle 抛出异常并判定 BUILD FAILED 。
根源剖析： 该命令本质上是让 Gradle 父进程分叉出一个子 JVM 进程来跑游戏，
且 Gradle 本身仍处于 EXECUTING 状态等待任务结束 。
此时，终端的 I/O 流被构建工具二次拦截与转码。当输入 Ctrl+Z（发送 EOF 信号）关闭输入流时，
引擎内部的 Scanner 抛出 NoSuchElementException 导致 JVM 崩溃，进而向 Gradle 传导了非零错误码，
最终导致构建判定失败 。
非耦合交付方案（Decoupled Deployment）：
废弃指令： 彻底抛弃 ./gradlew run 及其配套的强制转码和标准输入重定向的“狗皮膏药”代码。
规范化构建链路： 采用 ./gradlew :game-core:clean :game-core:installDist。
clean：物理摧毁之前被 Gradle 污染环境生成的 .class 毒化缓存。
installDist：进行纯净编译，自动收割外部依赖（Gson），
并在 build/install/game-core/bin/ 下生成自带正确 Classpath 映射的操作系统原生启动脚本（.bat）。
独立运行： 直接运行生成的 .bat 脚本，将控制权 100% 交还给操作系统的原生 JVM。
乱码与 I/O 冲突问题从物理层面不复存在。
